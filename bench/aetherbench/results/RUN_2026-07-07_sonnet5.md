## Fix-loop ablation (the headline experiment) — structured wins

Six failed first attempts entered the repair loop, both arms from the
same failed candidate, max 4 iterations:

| arm | fixed | fix@4 | mean iterations |
|-----|-------|-------|-----------------|
| structured (full diagnostic JSON incl. expected/actual) | 6/6 | 100% | 1.17 |
| prose (bare message + line only) | 5/6 | 83% | 1.00 (on fixed) |

Five failures were shallow (rename a variable, add an argument) — both
arms fixed those in one iteration; no separation. The separation came
from the one failure that required *seeing data*: t3_05 nl
(WRONG_OUTPUT). The structured arm received
`{"expected": "hello-fs\n", "actual": "('Ok', 'hello-fs')\n"}`, saw
the Result tuple, and fixed the unwrap by iteration 2. The prose arm
received only "the program ran but printed the wrong output" and
random-walked: guessed trailing-newline → added trim → runtime
AttributeError → removed trim → introduced a `result`-keyword parse
error → fixed the keyword → still wrong output at iteration 4.
Exhausted without converging.

Honest scope: n=6 failures, n=1 separating case. The direction matches
the diagnostics-as-API claim (docs/opus_4_7_architecture.md) but one
datapoint is an anecdote, not a measurement. The claim that IS
supported at n=6: structured feedback never does worse, and it is the
only arm that solved every failure.
