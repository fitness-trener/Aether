What this proves — and what it never will

Proves today (decidable)

  • The capability surface of every function — what it can reach, transitively through the call graph.
  • Effect composition: a caller must declare every effect its callees perform.
  • Whether a module's reachable code stays inside its declared capability boundary.
  • Exactly where it cannot decide — shown as UNPROVABLE, never hidden.

Will never do

  • Find logic bugs or wrong algorithms — undecidable in general (Rice's theorem).
  • Judge architectural or design quality.
  • Tell you whether the code does the right thing — only what it can reach.
  • Guess intent. It proves boundaries; humans own meaning.

Can extend (decidable only)

  • More decidable boundary properties: resource limits, data-flow / PII boundaries, temporal effect ordering.
  • More source languages (e.g. TypeScript) behind the same surface.
  • Hybrid static + runtime checks for what static analysis must leave UNPROVABLE.
  • Cross-file module composition as a real unit.
Known gaps in this build: analysis is single-file (one AST); calls through function-valued parameters and let-bound functions are reported as UNPROVABLE rather than assumed safe; programs with no module declaration are treated as an implicit all-capability grant and prove nothing until a module governs them.
On Python specifically: soundness holds for the statically-analyzable fraction. A call is treated as capability-free ONLY when it is a local function, a known-pure builtin, or a mapped import; every other call — unmapped imports, methods on values whose type is not tracked, and dynamic constructs (eval, exec, getattr, importlib) — is flagged UNPROVABLE, never assumed clean. A VIOLATION is a sound positive (the capability really is reachable) but not a complete inventory: more capability can hide behind an UNPROVABLE region in the same function. Strict mode is the honest floor; pragmatic mode additionally trusts an allowlist of unambiguously-pure builtin-type methods (excluding capability homonyms like get/read/execute) and is sound on tested code but not guaranteed sound in general.
Every claim on this page maps to a diagnostic the Aether engine emitted over your exact source — including each step of the containment loop, which is a live re-proof of a real source revision, not a script. Regions the engine cannot decide are shown as UNPROVABLE, never hidden to inflate coverage. The roadmap extends only to other decidable boundary properties (resource limits, data-flow boundaries, effect ordering) and more languages — never toward undecidable bug classes.
